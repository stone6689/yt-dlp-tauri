# Toolchain Update Pipeline Design

**Status:** Approved

**Date:** 2026-07-11

## Summary

Replace the yt-dlp-only updater with a reviewed, policy-driven toolchain pipeline for yt-dlp, Deno, FFmpeg, and FFprobe. GitHub Actions discovers upstream releases, opens one weekly pull request, validates the candidate toolchain on every supported platform, and publishes it only after human merge and validation of the exact `main` commit.

Toolchain releases are independent from application releases. Once the client has migrated to the dedicated `toolchain-stable` channel, routine tool updates do not require a new application version.

## Problem

The current automation has four disconnected behaviors:

- `update-yt-dlp.yml` updates only yt-dlp.
- `toolchain-freshness.yml` checks every source URL but cannot repair FFmpeg or Deno entries.
- `tools-manifest.json` is uploaded only when an application version is released.
- The client reads the manifest from the latest application release.

This leaves the repository and released manifest out of sync and makes Windows installation depend on short-lived FFmpeg autobuild URLs. A daily FFmpeg build can be pruned upstream even while the pinned binaries remain compatible.

## Goals

- Keep yt-dlp, Deno, FFmpeg, and FFprobe fresh under tool-specific policies.
- Create one human-reviewed toolchain pull request per week.
- Create an immediate emergency pull request when a released source becomes unavailable or a maintainer declares a security update.
- Validate the complete candidate toolchain on Windows x64, macOS Intel, and macOS ARM64 before publication.
- Exercise yt-dlp, Deno, FFmpeg, and FFprobe together, not only as independent executables.
- Publish toolchain revisions without changing the application version.
- Mirror volatile FFmpeg archives under project control after compatibility and license checks.
- Keep the currently working client toolchain active until a replacement is fully installed and verified.
- Preserve auditable provenance from upstream release assets to the published manifest.

## Non-Goals

- Automatically merge toolchain pull requests.
- Treat real video-site availability as a blocking release signal.
- Track FFmpeg daily autobuilds.
- Mirror yt-dlp or Deno while their immutable official release URLs remain available.
- Add Windows ARM64 support as part of this work.
- Redesign the application update workflow or general Settings interface.

## Approved Decisions

| Area | Decision |
| --- | --- |
| Update policy | Tool-specific freshness rules |
| Pull requests | One combined weekly PR with human merge |
| Emergencies | Immediate separate PR with the same blocking validation |
| Publication | Dedicated `toolchain-stable` release channel |
| FFmpeg | Mirror volatile archives after validation |
| Blocking tests | Deterministic cross-platform compatibility suite |
| Real-site tests | Non-blocking Canary with deduplicated alerts |
| Publication authority | A merged PR plus validation of the exact `main` commit |
| Client installation | Transactional activation with rollback |

## System Architecture

```text
Upstream GitHub releases
          |
          v
Weekly discovery / daily health check
          |
          v
bot/toolchain-weekly or bot/toolchain-emergency-<tool>
          |
          v
Toolchain pull request
          |
          v
Read-only cross-platform validation
          |
          v
Human merge
          |
          v
Validation on the exact main commit
          |
          v
toolchain-stable publication
          |
          v
Client check, staged install, and activation
```

The desktop client never resolves "latest" versions from upstream projects. It consumes only the manifest promoted by this repository.

## Repository Data Model

### `toolchain-policy.json`

This file is the manually maintained policy source. It defines:

- Upstream repository and API endpoint.
- Allowed release status and tag format.
- Target-to-asset selectors.
- Update cadence and selection rule.
- Approved download hosts.
- Whether an asset must be mirrored.
- Archive member paths and expected executable names.

Policy changes require normal code review and cannot be generated by the update bot.

### `toolchain-lock.json`

This file records the exact resolved candidate. Each asset entry includes:

- Upstream release ID, tag, and publication time.
- Upstream asset ID, name, immutable URL, size, and SHA-256 digest.
- Target platform and archive format.
- Extracted executable SHA-256 values.
- Final project mirror filename when mirroring is enabled.
- Toolchain revision and generation time.

The lock is the CI provenance record and the input to manifest generation.

### `src-tauri/tools-manifest.json`

This remains the runtime contract consumed by the application. It is generated from the lock and must not be hand-edited. The next schema adds a monotonic `revision` in `YYYYMMDD.N` form while retaining `retrievedAtUtc` for existing clients. `N` is the one-based sequence of generated candidates on that UTC date.

The published manifest is byte-for-byte identical to the checked-in manifest. Commit SHA, workflow run ID, and test details live in the validation report so publication does not mutate the reviewed runtime contract.

### Development Restore Script

`scripts/download-tools.ps1` reads `src-tauri/tools-manifest.json` instead of embedding duplicate URLs and hashes. This removes a source of drift from future update pull requests.

### Toolchain Changelog

Tool-only revisions are recorded in `TOOLCHAIN_CHANGELOG.md`. Application `CHANGELOG.md` remains reserved for application releases.

## Version Selection Policy

### yt-dlp

- Check weekly.
- Select the newest non-draft, non-prerelease GitHub release.
- Match the official Windows standalone executable and macOS standalone executable.
- Require GitHub asset SHA-256 digests.
- Create an emergency candidate when the currently published asset is unavailable.

### Deno

- Check weekly.
- Select the newest non-draft, non-prerelease GitHub release.
- Resolve the exact asset for each supported target.
- Require GitHub asset SHA-256 digests.
- Create an emergency candidate when the current asset is unavailable or a maintainer manually declares a compatibility or security update.

### FFmpeg and FFprobe

- Select the last successful autobuild from the previous complete UTC calendar month.
- Never use the mutable `latest` asset URL.
- Treat FFmpeg and FFprobe from the same archive as one source unit.
- Compute and record separate hashes for the extracted FFmpeg and FFprobe executables.
- Allow a reviewed policy override when a yt-dlp compatibility requirement needs a newer build.
- Mirror binary archives classified as volatile. Initially this covers the Windows win64 GPL archive from `yt-dlp/FFmpeg-Builds`.

The previous-month rule selects an upstream build that is retained by the upstream monthly retention policy while avoiding daily PR churn.

## Discovery and Pull Request Lifecycle

### Weekly Discovery

`.github/workflows/toolchain-discover.yml` runs every Monday at `03:17 UTC` and supports manual dispatch. It:

1. Fetches release metadata with authenticated GitHub API requests.
2. Resolves releases through `toolchain-policy.json`.
3. Downloads candidate metadata and required checksum files.
4. Generates a candidate lock, runtime manifest, toolchain changelog entry, and PR summary.
5. Exits without changing GitHub state when the resolved lock matches the current lock.
6. Creates or updates `bot/toolchain-weekly` and its single open PR.

If the weekly PR remains open, the next run updates the same branch and PR. The bot uses force-with-lease only on its dedicated generated branch and never rewrites maintainer branches.

Every generated branch is regenerated from the latest `main`. If an emergency PR lands while the weekly PR is open, the next discovery run rebases the candidate logically by resolving all tools again from the new stable lock. A redundant emergency PR is closed only after its change is present in `main`.

### Emergency Discovery

`.github/workflows/toolchain-freshness.yml` runs daily at `04:41 UTC` and supports manual emergency dispatch with `tool` and `reason` inputs. It checks the currently published stable manifest and all of its source assets.

An unavailable source automatically creates or updates `bot/toolchain-emergency-<tool>`. Security and compatibility emergencies are maintainer-triggered because upstream projects do not expose a uniform machine-readable security-release signal.

Emergency PRs contain only the affected source unit and use the same validation and human merge gates as weekly PRs.

### Bot Authentication

The discovery workflows use a narrowly scoped GitHub App installation token with:

- Contents: read and write.
- Pull requests: read and write.
- Metadata: read.

A GitHub App token is required because pushes and pull requests created with the repository `GITHUB_TOKEN` do not reliably trigger downstream pull-request workflows. A long-lived personal access token is not used.

## Pull Request Contents

Every generated PR includes:

- Old and proposed versions for each tool and target.
- Upstream release dates, release IDs, and asset IDs.
- Official asset digests and extracted binary hashes.
- Mirror filenames and expected final project URLs.
- Release-note links.
- Compatibility matrix status and Canary status.
- A generated explanation when a tool is unchanged by policy.

The PR changes only generated toolchain state and its focused maintenance scripts or tests. It does not publish release assets.

## Compatibility Validation

`.github/workflows/toolchain-validate.yml` is both a pull-request workflow and a reusable workflow called by publication. It has `contents: read` and no secrets.

### Native Matrix

| Target | Runner |
| --- | --- |
| Windows x64 | `windows-latest` |
| macOS x64 | `macos-15-intel` |
| macOS ARM64 | `macos-15` |

Each runner validates only its native target. A final aggregate job named `toolchain-validation` is the required branch-protection check.

### Layer 1: Metadata and Supply Chain

- Confirm release ID, tag, asset ID, name, and size against the authenticated API response.
- Enforce approved HTTPS hosts and repository identities.
- Download the complete asset.
- Verify the official asset digest before extraction.
- Reject archive path traversal, unexpected symlinks, and missing expected members.
- Hash each extracted executable and compare it with the candidate lock and manifest.
- Treat caches only as performance hints and revalidate every cached byte.

Project mirror URLs predicted by a PR do not exist until publication. PR validation verifies the upstream source, mirror filename, and final manifest mapping. Publication verifies the final project URL after upload.

### Layer 2: Executable Probes

- Run `yt-dlp --version`.
- Run `deno --version`.
- Run `ffmpeg -version`.
- Run `ffprobe -version`.
- Verify the executable architecture and expected target.
- Verify that yt-dlp accepts the configured Deno runtime and FFmpeg location without warnings about missing tools.

### Layer 3: Deterministic Integration

The compatibility harness generates a short synthetic audio/video source with the candidate FFmpeg, produces a local DASH presentation, and serves it from a loopback Node HTTP server. Candidate yt-dlp must:

1. Read the local presentation.
2. Select separate audio and video formats.
3. Download both streams.
4. Invoke candidate FFmpeg to merge them.
5. Produce an output that candidate FFprobe reports as containing the expected audio and video streams.

The harness runs without cookies, geographic dependencies, or public websites.

### Layer 4: Application Compatibility

The Rust installer core is separated from the Tauri command adapter so both the application and a native `toolchain-smoke` test binary call the same download, extraction, hash, probe, and path-resolution code. The matrix uses this binary rather than a parallel installer implementation.

Each target also runs:

- `npm test`
- `npm run build`
- `cargo test --manifest-path src-tauri/Cargo.toml --lib`
- `cargo check --manifest-path src-tauri/Cargo.toml`

### Baseline and Diagnosis

The current stable toolchain is tested first. The complete candidate is tested second.

- Baseline failure classifies the run as infrastructure failure.
- Baseline success plus candidate failure classifies the candidate as incompatible.
- On candidate failure, diagnostic jobs replace one tool source unit at a time to identify the yt-dlp, Deno, or FFmpeg boundary.

The bot keeps the PR open, applies a compatibility-blocked label, and updates one persistent PR comment with the failing targets and run links.

## Real-Site Canary

The candidate workflow runs a small public-site metadata and simulated-download Canary with the candidate yt-dlp and Deno configuration. Canary failures are non-blocking because site availability depends on region, cookies, anti-bot behavior, and network conditions.

The stable Canary also runs daily at `05:53 UTC`. Three consecutive actionable failures for the same site and failure class create or update one deduplicated GitHub Issue. Actionable classes cover toolchain installation, metadata extraction, simulated download, and an unavailable reviewed target. Authentication challenges, rate limits, HTTP 412 responses, and transient network failures remain visible in the artifacts as environmental outcomes and do not advance the alert counter. A successful or environmental outcome closes a prior actionable alert, while a new actionable failure class updates the existing site Issue without waiting for another three-run threshold.

The reviewed `toolchain-canary.json` file contains the public URLs and expected operation for each Canary. Consecutive-failure counters are stored in a small `canary-state.json` workflow artifact; each run reads the newest prior state artifact and uploads the next state. Canary state never changes source files or release assets.

Canary logs redact cookies, authorization headers, query credentials, and local paths before upload.

## Publication Model

### Stable Channel

The project owns a fixed GitHub prerelease tagged `toolchain-stable`. Marking it as prerelease prevents it from replacing the latest normal application release.

It contains:

- `tools-manifest-<revision>.json`, immutable reviewed manifests.
- `toolchain-validation-<revision>.json`, immutable validation reports.
- Versioned mirrored FFmpeg archives.
- FFmpeg checksum, license, and source-provenance materials required by the redistribution policy.

Versioned assets are not overwritten. Initial policy does not delete old mirrored FFmpeg assets or revision manifests.

The release body contains one machine-readable channel record with the promoted revision, manifest asset name, and manifest SHA-256. Updating the release body is the atomic promotion operation: a failed API update leaves the previous channel record intact.

```text
<!-- toolchain-channel
{"schemaVersion":1,"revision":"20260711.1","manifest":"tools-manifest-20260711.1.json","sha256":"aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa"}
-->
```

The publisher preserves the human-readable release body around this single marker. The client rejects duplicate markers, malformed JSON, unknown channel schemas, invalid revisions, and invalid digests.

### Publish Trigger

`.github/workflows/toolchain-publish.yml` runs on a `main` push that changes the lock or runtime manifest. It uses a concurrency group with cancellation disabled.

For a forward promotion, the publisher verifies that the commit is associated with a merged toolchain PR and that the aggregate required check succeeded. A direct push to `main` cannot promote a new toolchain revision. Manual rollback follows the separate rollback path described below.

The workflow calls the reusable native validation matrix on the exact `main` commit. Publication starts only after the aggregate validation succeeds and its report contains:

- The current commit SHA.
- The exact manifest SHA-256.
- The exact lock SHA-256.
- Successful results for all required targets.

Only the publisher has `contents: write` access to releases.

### Publish Transaction

1. Confirm that the revision is newer than the currently promoted revision.
2. Upload any new versioned FFmpeg archive and compliance assets.
3. Download each uploaded archive from its final project release URL and verify its digest.
4. Run final source URL and extracted-file checks using the release URLs.
5. Upload the immutable revision manifest and validation report.
6. Download the immutable manifest and verify that it is byte-for-byte identical to the checked-in manifest.
7. Atomically update the release-body channel record to promote the revision.
8. Update the compatibility copy of `tools-manifest.json` on the latest normal application release.

Uploading assets before the channel record ensures an interrupted publication leaves clients on the prior complete revision. An archive uploaded by an interrupted run is an auditable orphan and can be removed manually after the failure is understood.

## FFmpeg Redistribution Gate

Mirroring an FFmpeg GPL binary makes this project responsible for its redistribution obligations. The publisher therefore requires:

- The applicable license notices.
- Exact upstream binary release and digest.
- Exact FFmpeg source revision and build-repository revision.
- Source and build provenance sufficient to retrieve the corresponding source.
- A documented source-availability path in `THIRD-PARTY-NOTICES.md` and the release assets.

If the required compliance materials cannot be resolved and verified, automated mirroring remains blocked. The resolver may generate an explicit upstream-fallback candidate that uses the approved monthly URL; that fallback must be visible in the PR summary and pass the normal availability and compatibility gates. The pipeline never silently changes mirror policy and never mirrors an asset with incomplete provenance.

## Client Channel and Migration

One application release is required to establish the durable channel and transactional installer. After that migration, routine toolchain revisions do not require application releases.

The new client fetches:

```text
GET /repos/Chlience/yt-dlp-tauri/releases/tags/toolchain-stable
```

The migrated client reads the machine-readable channel record from the release body, locates the named immutable revision manifest, verifies its advertised SHA-256, and then compares the revision with the active manifest. It does not depend on replacing a mutable release asset.

The bundled manifest remains an offline fallback.

For existing clients, the publisher also updates the manifest asset on the latest normal application release. Existing v0.1.11 clients can continue using their current release-based update path during migration.

## Transactional Client Installation

Managed tools are stored by revision under the target root. The client keeps an atomic active-revision state file rather than deleting the current tool directory before download.

Update sequence:

1. Fetch and validate the stable manifest.
2. Download all target assets into a revision staging directory.
3. Verify archive and extracted-file hashes.
4. Run local executable and combination smoke probes.
5. Move the complete revision into the managed revisions directory.
6. Atomically replace the active-revision state file.
7. Retain the previous active revision for rollback.

Install, update, and reinstall share this sequence. A failed operation leaves the active revision unchanged. Legacy flat tool directories remain readable until the first successful revision-based activation.

## Rollback

The publisher supports manual dispatch with a previously published revision. A rollback:

1. Loads the immutable revision manifest and validation report.
2. Verifies that every referenced asset still exists and matches its digest.
3. Re-runs the deterministic compatibility suite unless the incident requires an explicitly approved emergency promotion.
4. Atomically updates the release-body channel record to the selected revision.
5. Records the rollback reason and actor in the release notes and workflow summary.

Clients then receive the older revision through the normal toolchain update path. The application version remains unchanged.

## Security and Permissions

- Discovery uses the scoped GitHub App token only for its generated branches and PRs.
- Pull-request validation has read-only repository access and no secrets.
- Candidate URLs are checked against policy and GitHub asset identities before execution.
- Publication runs only from `main` and has release write access.
- Canary issue management is isolated in a workflow with `issues: write` and no release access.
- All third-party actions used by these workflows are pinned to reviewed commit SHAs.
- Logs and uploaded reports exclude tokens, cookies, authorization headers, and sensitive URLs.

## Failure Handling

| Failure | Result |
| --- | --- |
| Upstream API unavailable | Fail discovery; leave the existing PR and stable release unchanged |
| Missing or ambiguous asset | Block candidate generation |
| Official digest missing | Block candidate generation |
| Baseline matrix failure | Classify as CI infrastructure failure; do not judge candidate |
| Candidate matrix failure | Keep PR open and run per-tool diagnostics |
| Compliance data missing | Block mirroring and retain approved upstream source URL |
| Mirror upload failure | Keep prior stable manifest active |
| Channel record update failure | Keep the prior revision promoted and leave new assets unpromoted |
| Client fetch failure | Use the active cached manifest or bundled fallback |
| Client install failure | Keep the current active revision |
| Canary failure | Warn; open a deduplicated issue after three consecutive failures |

## Observability and Audit

Every validation emits `toolchain-validation.json` with:

- Toolchain revision.
- Tool versions and targets.
- Release and asset IDs.
- Official asset digests.
- Extracted executable hashes.
- Matrix runner images and architectures.
- Deterministic test results.
- Canary status.
- Commit SHA, manifest SHA, lock SHA, workflow run ID, and run URL.

The workflows also publish concise GitHub Step Summaries. The bot edits a single PR status comment rather than adding one comment per run.

## Test Strategy

### Unit Tests

- Release filtering and stable-version selection.
- Previous-complete-month FFmpeg selection.
- Asset selector uniqueness.
- Digest normalization.
- Lock and manifest generation.
- Revision comparison.
- Approved-host enforcement.
- PR summary generation.

All resolver tests use checked-in API fixtures.

### Contract Tests

- Workflow triggers and permissions.
- GitHub App use in discovery.
- Read-only PR validation.
- Exact-main validation before publication.
- Immutable manifest uploaded after mirrored assets and before channel promotion.
- Stable channel marked prerelease.
- Application release compatibility copy retained.

### Integration Tests

- Complete archive download and digest validation.
- Safe extraction and expected-member selection.
- Runtime manifest installation through the shared Rust installer core.
- Local DASH download and FFmpeg merge.
- FFprobe verification of the final media.
- Published project mirror download.
- Promoted immutable manifest identity check.
- Rollback dry-run against a historical revision.

## Migration Sequence

1. Add policy, lock, resolver, generator, and deterministic fixtures.
2. Generalize the current yt-dlp updater into weekly and emergency discovery modes.
3. Add the native compatibility harness and reusable validation workflow.
4. Repair the currently unavailable Windows FFmpeg source through the first reviewed toolchain PR.
5. Create the `toolchain-stable` prerelease and publish the first validated revision.
6. Add the publisher and dual-publish compatibility copy.
7. Release one application migration version with the dedicated channel and transactional installer.
8. Retire the yt-dlp-only update workflow after the generalized pipeline has completed one successful weekly cycle and one manual rollback dry-run.

## Acceptance Criteria

- A weekly run creates at most one toolchain PR and produces no PR when no policy-selected version changed.
- A dead stable source creates an emergency PR within one daily freshness cycle.
- Every supported target passes supply-chain, executable, deterministic integration, and project checks before merge and before publish.
- A candidate created by the bot triggers normal pull-request checks.
- Merging a validated PR publishes a new toolchain revision without changing the application version.
- The stable manifest is never promoted before all referenced mirrored assets are available and verified.
- The promoted immutable manifest and checked-in runtime manifest are byte-identical.
- Existing v0.1.11 clients can consume the compatibility manifest during migration.
- A failed client update leaves the active tools usable.
- A historical validated revision can be promoted through the rollback workflow.
- Real-site Canary failures do not block publication and produce one deduplicated alert after three consecutive failures.

## Rejected Alternatives

### Track Every Latest Release

FFmpeg publishes too frequently for useful human review. Daily updates would create PR noise and consume the full compatibility matrix without a corresponding user benefit.

### Use Mutable `latest/download` URLs

Mutable assets conflict with pinned SHA-256 verification and make historical manifests unreproducible.

### Publish Directly From Discovery

Discovery has not passed human review or exact-main validation. Combining discovery and publication would broaden permissions and make the merge boundary meaningless.

### Auto-Merge Passing Toolchain PRs

Compatibility tests cannot cover every extractor, site, or operating environment. Human merge remains the explicit publication authorization.

### Block on Real Websites

Public-site tests are influenced by geography, cookies, anti-bot systems, and transient network behavior. They provide useful Canary evidence but are unsuitable as deterministic release gates.

### Wait for the Next Application Release

Tool freshness has a different cadence from application changes. Coupling both channels causes unnecessary application releases and leaves users on stale tool manifests between releases.
